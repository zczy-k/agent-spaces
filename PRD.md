
# 多 Agent 协同编程 Workspace：需求澄清与文档补全

## 1. 项目目标

**目标**：构建一个基于 **Next.js + shadcn/ui + TailwindCSS** 前端与 **Express + WebSocket (WSS)** 后端服务的多 Agent 协同编程平台，支持以下功能：

- **本地工作空间**：每个工作空间可以绑定多个本地代码目录，但仅认领一个 `.agentspace` 目录。
- **多 Agent 协作**：通过调度者、策划者、执行者和审核者 Agent 实现任务的分发、执行和审核。
- **多种工具集成**：包括代码编辑、终端调试、Git 操作等。
- **执行记录管理**：通过 `share/{议题id}` 目录存储共享执行记录。

## 2. 核心概念

### 2.1 Workspace

每个 workspace 是系统的最小管理单元，它包含以下内容：

- **绑定目录**：每个 workspace 可以绑定多个本地代码目录，但每个 workspace 仅认领一个 `.agentspace` 目录。
- **文件存储**：`.agentspace` 是每个 workspace 的元数据存储位置，包含技能文件、Agent 配置、claude.md 等。
- **任务与协作**：一个 workspace 下可以有多个频道、议题和 Agent（如调度者、策划者、执行者、审核者）。

### 2.2 `.agentspace` 目录

每个 workspace 有独立的 `.agentspace` 目录，包含以下内容：

- **skills**：技能库（自动生成并更新，包含默认的 `claude.md` 文件）。
- **agents**：Agent 配置。
- **claude.md**：初始化文件，包含默认的知识模型。
- **tasks**：任务管理。
- **cache/locks**：缓存和锁管理。
- **logs**：执行记录。

需要确保新的 Git 分支中包含 `.agentspace` 文件夹，并且只有策划者可以更新此文件夹的内容。

### 2.3 `share/` 目录

`share/` 目录存放所有 Agent 执行记录，并按议题 ID 隔离：

- 每个 Agent 保存独立的执行记录文件，格式为 MD，便于阅读。
- 执行记录支持版本控制，历史记录会被封存并移动到 `archive/` 目录中。

## 3. Agent 角色与职责

### 3.1 调度者（Scheduler）

- **作用**：定时检查服务器状态，如果空闲且有未完成的议题，唤醒策划者 Agent。
- **周期**：可以配置调度周期。
- **功能**：调度器负责唤醒任务，而不处理其他事务。

### 3.2 策划者（Planner）

- **作用**：管理议题、任务拆解、Agent 创建与任务分发。
- **职责**：
  - 不直接读写代码，负责创建 PR 和任务分发。
  - 根据执行者反馈更新任务并决定是否继续执行。
  - 策划者可访问整个 workspace 目录，且需要根据执行结果更新文档。

### 3.3 执行者（Executor）

- **作用**：执行代码分析与修改，完成任务。
- **限制**：
  - 只能使用基础命令工具（如 `grep`, `read`, `edit`）。
  - 有沙箱限制，不可访问除指定目录外的文件。
  - 执行结果成功后，通过 Hook 通知策划者。

### 3.4 审核者（Reviewer）

- **作用**：审核代码修改，确保符合质量标准。
- **职责**：
  - 阅读 diff，检查代码质量、风险等。
  - 提供审核结论：通过、修改、拒绝。
  - 审核完毕后通过 Hook 通知策划者，策划者根据审核结果决定下一步操作。

## 4. Agent Hook 机制

### 4.1 执行者完成 Hook

- 执行者任务完成后，如果成功，会启动一个新的会话，注入提示词与工具，写入执行结果并通知策划者。
- 执行记录格式为 Markdown。

### 4.2 策划者接收执行者通知 Hook

- 策划者在接收到执行者完成通知后，调用审核者进行代码审核。
- 审核者审核完后通过 Hook 通知策划者，策划者更新议题状态，决定是否继续修复或提交 PR。

## 5. 页面与功能模块

### 5.1 频道聊天

- 每个 workspace 可以创建多个频道，频道支持：
  - @mention Agent
  - 支持附件
  - 支持聊天记录搜索
  - 支持引用文件/代码片段

### 5.2 议题中心

- 每个 workspace 可以创建多个议题，支持：
  - 创建模糊议题，通过 @agent 请求详细 research。
  - research agent 自动生成计划文件，计划文件需标准化为 `skill.md` 格式。
  - 用户可以查看、编辑计划文件。

### 5.3 代码编辑

- 使用 Monaco Editor 实现代码编辑功能，支持：
  - 左侧文件树，右侧编辑 tab。
  - 支持文件夹浏览、文件修改、Git 操作等。

## 6. 界面布局

### 6.1 布局概述

系统界面包括：
- **workspace tabs**：顶部 tab 列表，管理不同 workspace。
- **left panel**：左侧面板，展示频道聊天、议题中心、代码编辑等内容。
- **right panel**：右侧面板，展示当前内容的详细信息，如聊天记录、代码、议题详情等。
- **bottom dock**：底部工具面板，包括终端、Codex、Claude Code、VSCode 调试、Git。

### 6.2 终端与 Git 操作

- 终端支持拖拽调整大小，支持多开。
- Git 面板支持查看 diff（暂时），后续可以扩展支持更多 Git 操作，如 commit、push、pull 等。

## 7. 存储与文件系统

### 7.1 文件存储设计

- 系统只使用本地文件系统，支持 Windows、macOS、Linux。
- 文件操作不做白名单限制，允许修改所有文件，但不提供撤销/回滚功能。
- 执行记录采用 Markdown 格式，更易阅读。
- 需要生成 Git diff 文件来跟踪文件变化。

## 8. 事件与状态机设计

### 8.1 议题状态

定义以下议题状态：
- `draft`
- `planned`
- `in_progress`
- `review_pending`
- `changes_requested`
- `approved`
- `completed`
- `archived`
- `error`

### 8.2 任务状态

定义以下任务状态：
- `pending`
- `running`
- `waiting_review`
- `retrying`
- `done`
- `failed`
- `cancelled`

### 8.3 Agent 会话状态

定义以下 Agent 会话状态：
- `idle`
- `active`
- `blocked`
- `completed`
- `crashed`

## 9. 数据层设计

### 9.1 数据存储格式

- 当前数据使用 JSON 格式存储，后续可以扩展为更高效的数据库存储。
- 目前不需要实现数据索引和搜索功能，未来根据需求再扩展。
- 不需要自动备份或版本历史。

# 10. 关键需求问题

### 10.1 产品层
1. **是否支持多人协作**：目前不支持多人协作。
2. **是否需要登录/账号体系**：不需要，Web端和后端分离，单一用户使用。
3. **是否需要 GitHub/GitLab 集成**：是，PR 会同步到 GitHub/GitLab，自动创建分支并提交 commit。

### 10.2 文件系统层
1. **是否支持文件修改**：允许修改所有文件，但不做危险操作保护。
2. **是否需要撤销/回滚**：不需要。
3. **是否支持 Git diff 生成**：是，支持生成 Git diff 跟踪文件变化。
